Semantic Git Branching Strategy
==============================================================================
A common challenge is that the CI system may not have the capability to run a sequence of build jobs conditionally. For instance, we may not always want to build the Lambda layer, which can be both time-consuming and unnecessary. Similarly, we may not want to run integration tests for small feature improvements made through a simple pull request.

In this project, we have adopted the semantic git branching convention, using different branch names for specific purposes. The CI system relies on these branch names to determine which jobs should be executed and which can be skipped.

Below is the detailed CI/CD build job workflow that runs only on specific Git branches. Each column header represents a semantic branch name, it follows the convention ``${du_name}/${semantic_name}/${description}``. For example, ``simple_sfn/feature/add-an-awesome-feature`` indicates a feature branch. The row index denotes the workflow action.

- On the ``main`` branch, the CI job is not triggered.
- On ``feature`` and ``fix`` branches, it triggers a CodeBuild CI job that performs code coverage tests (unit tests) only.
- On the ``doc`` branch, in addition to code coverage tests, it also builds the documentation website and deploys it to AWS S3.
- On the ``layer`` branch, in addition to code coverage tests, it also builds and publishes the Lambda dependency layer.
- On the ``ecr`` branch, in addition to code coverage tests, it also builds and publishes the custom Lambda container image.
- On the ``lambda`` branch, in addition to code coverage tests, it deploys the CDK stack along with the Lambda functions. Afterward, it runs integration tests.
- On the ``release`` branch, it triggers a CodePipeline execution. This pipeline repeats the behavior of the ``lambda`` branch in the ``sbx`` and ``tst`` environments. It waits for manual approval before deploying to the ``prd`` environment.
- On the ``cleanup`` branch, if the commit message matches the predefined pattern ``cleanup: ${env_name_1}, ${env_name_2}, ...``, it will delete the CDK stack, API gateway, and Lambda function in the specified environment.
